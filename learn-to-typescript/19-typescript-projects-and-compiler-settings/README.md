# Module 19: TypeScript Projects and Compiler Settings

## Before you start

Complete Modules 01 to 18. You should understand modules and the difference between checking and runtime behavior.

## What you will learn

- understand the roles of `package.json` and `tsconfig.json`
- install project-local tools
- check without emitting and build JavaScript output
- configure a strict Node ESM project
- understand several settings that reveal real missing-data risks
- use repeatable package scripts

## Why this matters

A project configuration makes every file follow the same rules. Commands become repeatable, editor behavior matches the command line, and the emitted JavaScript matches the intended runtime.

## 1. The project files have different jobs

`package.json` describes the JavaScript package. It records scripts, dependencies, and runtime details such as whether Node treats `.js` files as ECMAScript modules.

`tsconfig.json` describes a TypeScript project. It tells TypeScript which files belong to the project, how strictly to check them, how to resolve modules, and where to place output.

These files work together but are not interchangeable.

## 2. Create the project

```text
mkdir typed-reading-list
cd typed-reading-list
npm init -y
npm install --save-dev typescript tsx @types/node
mkdir src
```

- `typescript` provides the `tsc` compiler and type checker.
- `tsx` runs TypeScript conveniently during development.
- `@types/node` describes Node.js APIs for the type checker.
- `--save-dev` records tools needed for development rather than runtime application logic.

Use a local TypeScript dependency so the project controls its compiler version. Run it through `npx` or package scripts.

## 3. Configure `package.json`

Keep the fields generated by `npm init`, then set `type` and scripts. A minimal relevant shape is:

```json
{
  "name": "typed-reading-list",
  "version": "1.0.0",
  "type": "module",
  "scripts": {
    "check": "tsc --noEmit",
    "build": "tsc",
    "dev": "tsx src/index.ts",
    "start": "node dist/index.js"
  },
  "devDependencies": {
    "@types/node": "installed-version",
    "tsx": "installed-version",
    "typescript": "installed-version"
  }
}
```

Do not replace installed version numbers with the words shown above. `npm install` writes real versions for you.

`"type": "module"` tells Node that emitted `.js` files use ECMAScript module syntax.

## 4. Create `tsconfig.json`

```json
{
  "compilerOptions": {
    "target": "ES2022",
    "module": "NodeNext",
    "moduleResolution": "NodeNext",
    "rootDir": "./src",
    "outDir": "./dist",
    "strict": true,
    "noUncheckedIndexedAccess": true,
    "exactOptionalPropertyTypes": true,
    "verbatimModuleSyntax": true,
    "noEmitOnError": true,
    "sourceMap": true,
    "types": ["node"]
  },
  "include": ["src/**/*.ts"]
}
```

JSON uses double quotation marks and does not allow ordinary comments in every tool. `tsconfig.json` itself supports comments, but this course keeps the base file plain.

## 5. Understand each setting

### `target`

Controls the JavaScript language level TypeScript emits. `ES2022` is a reasonable modern Node target for this course. Choose a target supported by the actual runtime.

### `module` and `moduleResolution`

`NodeNext` models Node's modern ESM and CommonJS rules. The module resolver should match the runtime or bundler that loads the emitted JavaScript.

### `rootDir` and `outDir`

Source files live in `src`. Emitted JavaScript and source maps go to `dist`.

### `strict`

Enables a family of stronger checks, including strict null checking and checks against unclear implicit parameter types. A future TypeScript version may add or adjust checks under this family, so compiler upgrades can reveal new errors.

### `noUncheckedIndexedAccess`

Adds `undefined` to values read through unchecked array indexes and index signatures. This matches the runtime possibility that a key or index is missing.

### `exactOptionalPropertyTypes`

Distinguishes an absent optional property from a present property containing undefined unless undefined is explicitly included in the value type.

### `verbatimModuleSyntax`

Keeps module syntax predictable and makes type-only imports explicit. Use `import type` for names used only as types.

### `noEmitOnError`

Prevents new JavaScript output when the project has type errors.

### `sourceMap`

Emits map files that help tools connect running JavaScript locations to TypeScript source locations.

### `types`

Limits automatically included global type packages here to Node. Imported packages still bring their own referenced types as needed.

## 6. Add source files

`src/math.ts`:

```typescript
export function add(left: number, right: number): number {
  return left + right;
}
```

`src/index.ts`:

```typescript
import { add } from "./math.js";

console.log(add(2, 3));
```

## 7. Use the scripts

Check without creating JavaScript:

```text
npm run check
```

Run TypeScript directly during development:

```text
npm run dev
```

Build the project:

```text
npm run build
```

Run emitted JavaScript:

```text
npm start
```

After building, `dist/index.js` and `dist/math.js` should exist.

## 8. Checking and running are separate

`tsx` focuses on running TypeScript conveniently. Do not assume that a successful development run proves the whole project passes type checking.

Use `npm run check` as the clear project-wide type check. Use `npm run dev` for quick execution. Use `npm run build` to create distributable JavaScript.

## 9. Do not copy a configuration blindly

Browser apps, Node apps, libraries, and bundler projects need different module and library settings. A framework may generate and manage its own configuration.

Begin with the runtime and build tool. Then choose TypeScript settings that model them. More flags are not automatically safer if they do not match the environment.

## Common mistakes

### Installing TypeScript globally only

A project-local version makes builds repeatable across machines.

### Mixing Node ESM and CommonJS settings

Keep `package.json`, imports, compiler module settings, and the runtime aligned.

### Running code but never checking the project

Use a separate `check` script.

### Ignoring new errors after enabling indexed access checks

The errors often reveal real missing index or dictionary cases. Handle them instead of hiding them.
