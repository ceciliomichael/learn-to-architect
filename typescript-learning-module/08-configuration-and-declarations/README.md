# Module 08: Configuration, Modules, and Declaration Files

Writing TypeScript in isolation is one thing. Wiring it up as a real project with proper compiler settings, external libraries, and multiple files is another. This module covers the practical setup knowledge you need for real-world TypeScript development.

---

## 1. How TypeScript Projects Are Structured

A standard TypeScript project has:
- A `tsconfig.json` file: tells the compiler how to behave.
- A `src/` folder: where all your `.ts` source files live.
- A `dist/` folder: where the compiled `.js` output files go.
- A `package.json` file: manages your project dependencies (libraries you install).

```text
my-project/
├── src/
│   ├── index.ts
│   └── utils.ts
├── dist/             (generated by the compiler)
├── tsconfig.json
└── package.json
```

---

## 2. `tsconfig.json`: Configuring the Compiler

`tsconfig.json` is a JSON file at the root of your project that controls how `tsc` (the TypeScript compiler) behaves. Here is a standard configuration with every option explained:

```json
{
  "compilerOptions": {
    "target": "ES2020",
    "module": "CommonJS",
    "outDir": "./dist",
    "rootDir": "./src",
    "strict": true,
    "skipLibCheck": true,
    "esModuleInterop": true
  },
  "include": ["src/**/*"],
  "exclude": ["node_modules"]
}
```

### What each option means:

- **`target`**: The version of JavaScript to output. `ES2020` is modern and supported in all recent environments.
- **`module`**: How the compiled files export and import each other. `CommonJS` is the standard for Node.js.
- **`outDir`**: The folder where compiled `.js` files are placed.
- **`rootDir`**: The folder where your TypeScript source files live.
- **`strict`**: Enables all strict safety checks at once. Always set this to `true`. Explained in detail below.
- **`skipLibCheck`**: Skips type-checking inside `node_modules`. This makes compilation faster.
- **`esModuleInterop`**: Allows clean `import` syntax for certain older JavaScript libraries.
- **`include`**: A pattern telling the compiler which files to compile. `src/**/*` means everything inside `src/`.
- **`exclude`**: Files or folders to ignore.

---

## 3. What `"strict": true` Actually Enables

When you set `"strict": true`, it activates a collection of individual stricter checks all at once. The most important one is `strictNullChecks`.

### `strictNullChecks`

Without this check, `null` and `undefined` can be silently assigned to any variable typed as `string`, `number`, etc. This is the single biggest source of runtime crashes in JavaScript programs ("Cannot read properties of null").

With `strictNullChecks` enabled:
```typescript
let name: string = null; // ERROR! Type 'null' is not assignable to type 'string'.
```

To allow a value to be null, you must explicitly say so:
```typescript
let name: string | null = null; // Valid. You explicitly acknowledge null is possible.
```

This forces you to handle the "no value" case deliberately everywhere in your code.

## 4. Modules: `import` and `export`

A module is any TypeScript file that uses the `import` or `export` keyword. Splitting code into modules is how you organize large projects into manageable files.

### Exporting

You mark a function, variable, interface, or class as public to other files using `export`:

```typescript
// utils.ts
export function formatDate(date: Date): string {
  return date.toISOString().split("T")[0];
}

export const MAX_RETRIES = 3;

export interface Config {
  host: string;
  port: number;
}
```

You can also collect all exports at the bottom of a file:

```typescript
// math.ts
function add(a: number, b: number): number { return a + b; }
function subtract(a: number, b: number): number { return a - b; }

export { add, subtract };
```

### Importing

You bring exported items into another file using `import`:

```typescript
// index.ts
import { formatDate, MAX_RETRIES } from "./utils";
import { add } from "./math";

console.log(add(1, 2));       // 3
console.log(MAX_RETRIES);     // 3
```

The path `"./utils"` means "the file named `utils.ts` in the same directory as this file." The `./` prefix means "relative to this file's location."

### Default Exports

A file can have one default export, which does not need a name when imported:

```typescript
// logger.ts
export default function log(msg: string): void {
  console.log(msg);
}

// index.ts
import log from "./logger"; // No curly braces needed for default imports.
log("Hello");
```

## 5. Script Mode vs Module Mode

TypeScript treats files differently depending on whether they contain any `import` or `export` statements.

- **Script mode**: No `import` or `export`. Every variable declared in the file is global and shared across all files. If two files both declare `const myVar = 10`, TypeScript throws a "Duplicate identifier" error.
- **Module mode**: Has at least one `import` or `export`. Each file has its own private scope. Variables in one file do not conflict with variables in another.

The fix for unwanted script mode: add `export {};` at the top of any file that does not naturally export anything. This one line converts the file into a module:

```typescript
export {}; // Converts this file to module scope.

const myVar = 10; // Now private to this file only.
```

## 6. Declaration Files (`.d.ts`)

When you use a JavaScript library that was not written in TypeScript, that library has no type information. TypeScript does not know what functions or objects the library provides.

A **declaration file** (`.d.ts`) is a file that contains only type information, no executable code. It describes the types of a JavaScript library so TypeScript can understand it.

Declaration files:
- Only contain type signatures, interfaces, and declarations.
- Never contain function bodies or any executable logic.
- Are never compiled to `.js`. They exist purely for the TypeScript compiler.

Example: Typing a global object that a legacy JavaScript file adds to `window`:

```typescript
// analytics.d.ts
interface Analytics {
  init(apiKey: string): void;
  trackEvent(name: string, data?: Record<string, string>): void;
}

declare global {
  interface Window {
    Analytics: Analytics;
  }
}

export {}; // Makes this a module-scoped declaration file.
```

## 7. DefinitelyTyped and `@types` Packages

Most popular JavaScript libraries have community-maintained declaration files available through a project called **DefinitelyTyped**. You install them as development dependencies using the `@types/` prefix:

```bash
npm install lodash
npm install -D @types/lodash
```

After installing, TypeScript can understand `lodash`'s functions and give you full autocompletion and type checking:

```typescript
import _ from "lodash";

let shuffled = _.shuffle([1, 2, 3, 4, 5]); // TypeScript knows this returns number[].
```

Some modern libraries (like `axios`, `zod`, `prisma`) include their own TypeScript types built in, so you do not need a separate `@types/` package.

## 8. Path Aliases (Cleaner Imports)

As projects grow, relative import paths like `"../../../utils/formatter"` become hard to read. You can configure path aliases in `tsconfig.json` to create shortcuts:

```json
{
  "compilerOptions": {
    "baseUrl": ".",
    "paths": {
      "@utils/*": ["src/utils/*"],
      "@types/*": ["src/types/*"]
    }
  }
}
```

After this configuration, you can write:

```typescript
import { formatDate } from "@utils/formatter"; // Instead of "../../utils/formatter"
```

## 9. The `npm` Ecosystem: Installing and Managing Libraries

`npm` (Node Package Manager) is the tool you use to install external libraries into your TypeScript project.

Common commands:
```bash
npm init -y               # Creates a package.json file for your project.
npm install typescript    # Installs TypeScript as a dependency.
npm install -D typescript # Installs TypeScript as a dev-only dependency (recommended).
npx tsc --init            # Creates a starter tsconfig.json file.
npx tsc                   # Compiles your project using tsconfig.json.
npx tsc --watch           # Re-compiles automatically every time you save a file.
```

The `-D` flag means "development dependency." Libraries installed with `-D` are only needed while building. They are not shipped in the final production code.
